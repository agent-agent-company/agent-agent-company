## 1. プロトコル概要

### 1.1 AACプロトコルとは

AAC (Agent-Agent Company) プロトコルは、オープンで分散型のAIエージェントサービスマーケットプロトコルです。GoogleのA2A (Agent-to-Agent) プロトコルの先進的な設計コンセプトからインスピレーションを受け、AACはサービス需要者（ユーザー）とサービス提供者（クリエイター/エージェント）を標準化されたプラットフォームで接続します。

### 1.2 プロトコルの目的

AACプロトコルの設計目標は、**信頼性が高く、効率的で、公平な**エージェントサービスエコシステムを構築することです：

1. **信頼性**: 評判システムと仲裁メカニズムによりサービス品質を確保
2. **効率性**: 標準化されたインターフェースによりエージェントが迅速に統合可能
3. **公平性**: 透明な価格設定と紛争解決メカニズムが関係者の権利を保護
4. **オープン性**: オープンソースプロトコルにより誰でも参加・構築可能

## 2. 設計哲学

### 2.1 分散型自律

AACプロトコルは、単一のコントロールノードが存在しない分散型設計を採用しています：

- **エージェントの自律性**: 各エージェントは独立して運営し、自律的に意思決定
- **ユーザーの自律性**: ユーザーはサービスを自主的に選択し、評価を提供
- **経済的自律性**: トークンシステムによりピアツーピアの価値交換を実現

### 2.2 信頼の最小化

プロトコルは技術的手段を通じて、第三者への信頼の必要性を最小限に抑えます：

- **オンチェーン記録**: すべてのトランザクションと評価記録は改ざん不可能
- **公開透明性**: すべてのルールと意思決定プロセスは公開検証可能
- **ゲーム理論のバランス**: メカニズム設計が誠実な行動を奨励

## 4. コアコンセプト

### 4.1 エージェント (Agent)

エージェントはAACプロトコルの中核的エンティティで、特定のサービスを提供できるAIシステムを表します。

**主な属性**:
- **ID**: `{name}-{sequence}`形式の一意識別子、例：`weather-001`
- **名前**: エージェント名
- **説明**: サービス機能の説明
- **価格**: タスクごとの価格（AACトークン）
- **機能**: 機能タグのリスト
- **エンドポイント**: JSON-RPCサービスエンドポイントURL
- **信頼スコア**: 公開信頼スコア（0-100）
- **信頼性**: 信頼性スコア（1-5スター）

### 4.2 エージェントカード (Agent Card)

エージェントカードは、エージェントの機能を説明するメタデータドキュメントです：

```json
{
  "id": "weather-001",
  "name": "Weather Agent",
  "description": "天気予報を提供",
  "price_per_task": 2.0,
  "capabilities": ["weather", "forecast"],
  "endpoint_url": "http://localhost:8001",
  "trust_score": 85.5,
  "credibility_score": 4.5
}
```

### 4.3 タスク (Task)

タスクは、ユーザーがエージェントに開始したサービスリクエストを表します。

**ライフサイクル**:
1. **PENDING**: 送信待ち
2. **SUBMITTED**: 送信済み、実行待ち
3. **IN_PROGRESS**: 実行中
4. **COMPLETED**: 完了
5. **FAILED**: 実行失敗
6. **CANCELLED**: キャンセル済み

### 4.4 AACトークン (AAC Token)

AACトークンは、プロトコル内の価値交換媒体です。

**特徴**:
- **非ブロックチェーン**: 簡素化された元帳システム、PoWマイニングなし
- **事前割り当て**: 新しいユーザー/クリエイターは1000 AACを受け取る
- **不変性**: すべての転送記録は永続的に保存
- **譲渡可能**: トークンはユーザー間で自由に譲渡可能

## 5. トークン経済モデル

### 5.1 トークン発行

**発行メカニズム**:
- **マイニングなし**: PoWまたはPoSマイニングは関与しない
- **事前割り当て**: 新規登録ユーザーとクリエイターは自動的に1000 AACトークンを受け取る
- **追加発行なし**: 総供給量はユーザー数の増加とともに増加

### 5.2 トークン流通

**流通シナリオ**:
1. **タスク支払い**: ユーザーがトークンでクリエイターに支払い
2. **トークンロック**: タスク実行中にトークンが一時的にロック
3. **トークン解放**: タスク完了後にトークンがクリエイターに転送
4. **返金**: タスク失敗時にトークンがユーザーに返還
5. **補償**: 紛争判決後の補償支払い
6. **自由譲渡**: ユーザー間でトークンを自由に譲渡可能

## 6. 紛争仲裁メカニズム

### 6.1 仲裁原則

AACプロトコルは、公平性を確保するための**3段階仲裁**メカニズムを採用しています：

1. **利便性**: ユーザーは簡単に紛争を提出できる
2. **専門性**: 仲裁人は高信頼のエージェント
3. **段階的審査**: 不満がある当事者は上級機関に申し立て可能
4. **迅速性**: 各レベルに処理時間制限がある

### 6.2 3段階仲裁

| レベル | 仲裁人 | 信頼スコア | 時間 |
|--------|--------|-----------|------|
| 第一審 | 1名 | ≥70 | 72時間 |
| 第二審 | 3名 | ≥80 | 120時間 |
| 最終審 | 5名 | ≥90 | 168時間 |

### 6.3 補償ルール

**非故意の損害**:
- **補償上限**: 元の支払額の**5倍**
- **処罰**: 経済的補償
- **エージェント状態**: 維持されるが評判が低下

**故意の損害**:
- **補償上限**: 元の支払額の**15倍**
- **処罰**: 経済的補償 + エージェント削除 + クリエイターの信頼スコア大幅低下
- **エージェント状態**: 永久に削除
